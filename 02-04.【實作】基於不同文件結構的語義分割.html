🎯 Chunking 的核心目標

Chunking 的目的是讓每一個 Chunk 都擁有獨立、清晰的主題。Chunk 切得好，後續的向量檢索才能精準命中正確的參考資料。

本節使用 LangChain Splitter 示範四種不同的分割策略，以「珍珠奶茶的歷史起源與產業發展」作為範例文本。

⚙️ 所有策略都需掌握的兩個核心參數

chunk_size
每個 Chunk 的最大長度（字元數或 Token 數）。決定每段資料能涵蓋多少內容。
💡 設太大：語意混雜，檢索不精準
💡 設太小：上下文不足，難以回答完整問題
chunk_overlap
相鄰兩個 Chunk 之間「重疊」的長度，用來保持跨 Chunk 的上下文一致性。
💡 Overlap 讓邊界處的句子不會被切斷失去語意
💡 過大的 Overlap 會增加資料庫儲存量
Chunk Overlap 示意：相鄰 Chunk 共享一段重疊區域以保留上下文
Chunk 1 主體
← Overlap →
Chunk 2 主體
Overlap 區域在兩個 Chunk 中都存在，確保銜接處語意完整

🧩 四種 Chunking 策略詳解

01
CharacterTextSplitter
固定長度字元切割
最基礎 · 最快速

最直觀的切割方式，依照設定的 chunk_size 直接切斷文字，不考慮語意邊界。適合對切割精準度要求不高、快速原型驗證的場景。也支援以 Token 為單位切割（先 Embedding 再切）。

Python — CharacterTextSplitter
from langchain.text_splitter import CharacterTextSplitter

splitter = CharacterTextSplitter(
    chunk_size=200,      # 每個 chunk 最多 200 字元
    chunk_overlap=20,   # 相鄰 chunk 重疊 20 字元
    separator=""        # 不依任何分隔符，純粹按長度切
)
chunks = splitter.split_text(text)
# 結果：文本依固定長度切成數段，中間有 overlap 區域
chunk_size chunk_overlap token-based 變體 ✓ 簡單快速 ⚠ 可能從句子中間截斷
02
RecursiveCharacterTextSplitter
遞迴字元切割
最常用 · 推薦預設

先嘗試以段落（\n\n）切，切不夠小再以句子（\n）切，再不夠再以標點符號（逗號、句號）切，由大切小、遞層細分，盡可能在自然語言的斷點處切割，保留語意完整性。

Python — RecursiveCharacterTextSplitter
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=200,
    chunk_overlap=20,
    separators=["\n\n", "\n", "。", "，", " ", ""]
    # 依優先順序嘗試這些分隔符，由大到小遞迴切割
)
chunks = splitter.split_text(text)
# 結果：在自然語言斷點處切割，語意保留比固定切割更好
separators 優先序 由大切小 中文友善 ✓ 最推薦的通用做法 ✓ 中英文皆適用
03
SemanticChunker ／ K-Means 分群
語意感知切割
語意最佳 · 成本最高

SemanticChunker：先將每兩個相鄰句子進行 Embedding，計算語意相似度，當前後兩句的語意出現重大落差（breakpoint）時即切斷。參數 breakpoint_threshold_type 可設為 percentile，控制多大的落差才算「重大轉折」。

K-Means 分群（土炮做法）：先將所有句子 Embedding，再用 K-Means 演算法按語意距離分成 K 群，每群即為一個 Chunk，效果近似 SemanticChunker 但不需依賴框架。

Python — SemanticChunker（LangChain）
from langchain_experimental.text_splitter import SemanticChunker
from langchain_openai import OpenAIEmbeddings

splitter = SemanticChunker(
    embeddings=OpenAIEmbeddings(),
    breakpoint_threshold_type="percentile",
    breakpoint_threshold_amount=70  # 語意落差 > 70 分位點才切斷
    # 數值越大 → 越難切 → 段落越少越大
    # 數值越小 → 越容易切 → 段落越多越細
)
chunks = splitter.create_documents([text])
Python — K-Means 分群（土炮語意切割）
from sklearn.cluster import KMeans
import numpy as np

# 先將每句話 Embedding
sentences = text.split("。")
embeddings = [get_embedding(s) for s in sentences]

# K-Means 分成 3 群
kmeans = KMeans(n_clusters=3, random_state=42)
labels = kmeans.fit_predict(np.array(embeddings))

# 依照 cluster 標籤合併同群句子 → 得到 3 個語意 Chunk
breakpoint_threshold percentile K-Means ✓ 語意切割品質最佳 ⚠ 中文效果有限，建議換 Embedding 模型 ⚠ 需呼叫 Embedding API，成本較高
04
RecursiveCharacterTextSplitter（Language Mode）
程式碼感知切割
程式碼專用

設定 language="python"（或其他語言）後，切割器會依據程式語言的語法特性——class、function、import 區塊、return 語句、註解等——作為分割邊界，而不是依字元長度強制截斷，使每個 Chunk 保留完整的程式碼邏輯單元。

Python — 程式碼感知切割
from langchain.text_splitter import RecursiveCharacterTextSplitter, Language

splitter = RecursiveCharacterTextSplitter.from_language(
    language=Language.PYTHON,   # 可換成 JS, TS, Go, Java, C++ 等
    chunk_size=500,
    chunk_overlap=50,
)
chunks = splitter.split_text(python_code)
# 結果切成：import 區塊、class 定義、function 定義、return 區塊…
Language.PYTHON Language.JS function / class 邊界 ✓ 程式碼 RAG 的最佳選擇 ⚠ 僅適用於程式碼文本

⚖️ Chunk 大小的三個區間

🔬
Chunk 太小
每段資訊片段化，缺乏完整語意，LLM 拿到的上下文不足以回答完整問題
⟺
✅
最佳區間
每個 Chunk 對應一個獨立主題，能精準被檢索，且 LLM 能從中讀取完整資訊
⟺
🌊
Chunk 太大
多個主題混雜在一個 Chunk，向量語意被稀釋，檢索精準度下降

🈶 中文語意切割的限制與因應

⚠ 語意切割（SemanticChunker）在中文的效果有限！

主因是大多數 Embedding 模型對中文的語意向量品質不如英文，導致前後句子的語意落差計算不夠精準，分割結果往往不如預期。

建議因應策略：
1. 改用支援中文的 Embedding 模型（如 text-embedding-3-small 搭配中文語料微調版）
2. 改用 RecursiveCharacterTextSplitter，搭配中文常用標點符號作為分隔符（句號、換行、逗號）
3. 嘗試 K-Means 分群，效果相對穩定且不依賴 Embedding 品質

💡 核心觀念

切割目標只有一個：每個 Chunk 擁有一個獨立主題。主題越獨立，向量越清晰，檢索越精準，最終生成的答案品質越好。
不同資料用不同策略：一般文章 → RecursiveCharacterTextSplitter；需要最佳語意品質 → SemanticChunker；程式碼 → Language Mode；快速原型 → CharacterTextSplitter。
Chunk 的設計是 RAG 系統中最容易被忽視、卻對最終品質影響最深的環節。前端資料若切不好，後面再怎麼優化 Retriever 與 LLM 也難以彌補。
