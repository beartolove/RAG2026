Chunk 的選擇對查準率的影響

張維元 ── RAG 精準搜尋實戰 02-06 | Chunk Size × Overlap × Embedding Model 三參數調教，從實驗找到可用解

🎯 核心觀念：Chunking 是被忽略的調參空間

大多數人優化 RAG 時只想到換模型、換 Embedding，卻忽略了 Chunking 本身就是一個重要的可調參數（Parameter）。

Chunk 的大小、重疊比例、切分邊界，本質上都在調整查準率（Precision）與查全率（Recall）的平衡。沒有做過實驗與評估，只是在盲測。

⚖️ Chunk Size 的根本取捨：查準 vs 查全

CHUNK SIZE 對檢索品質的雙面影響
🔬
Chunk 切太小
  • 語意集中，查準率高
  • 容易精準對應到特定問題
  • 資訊不完整，缺乏完整語境
  • 關鍵上下文可能被切斷
  • LLM 拿到的片段不足以回答
VS
🌊
Chunk 切太大
  • 上下文完整，不易誤解
  • 保留較完整的語意單元
  • Embedding 後向量模糊失真
  • 混入太多主題，重點被稀釋
  • 檢索相似度下降，命中率低
CHUNK SIZE 對兩個指標的影響趨勢
查準率（Precision）
小 Chunk → 高大 Chunk → 低
查全率（Recall / 上下文完整性）
小 Chunk → 低大 Chunk → 高
⚡ 最佳 Chunk Size：讓每個 Chunk 對應一個獨立主題，查準率與上下文完整性同時最大化

📏 依文件類型的 Chunk Size 建議值

SCENARIO 01
❓

FAQ 問答文件

300 – 500 Tokens

每個 FAQ 條目通常只有一問一答，語意單元本身短小精悍，3～4 句話大致就能涵蓋一個完整知識點。

💡 使用者懶得看長文，FAQ 本來就是濃縮過的資訊
SCENARIO 02
📘

技術文件 / API 說明

800 – 1000 Tokens

包含 API 說明、參數定義、範例程式碼，語意單元較長，需要保留完整的說明脈絡與前後依賴關係。

💡 切太小會把 API 說明和範例程式切斷，導致 LLM 無法理解完整用法
SCENARIO 03
📄

一般文章 / 報告

500 – 800 Tokens

段落結構較清晰，以自然段落為切割邊界效果最好。可搭配 RecursiveCharacterTextSplitter 依換行符切割。

💡 中文段落通常以換行或句號為自然邊界，優先利用這些斷點

🔗 Chunk Overlap 的建議區間

5%
文本乾淨時可用
文字結構清晰、段落完整，語意邊界自然，不需太多重疊
10 – 20%
✅ 一般建議區間
適合多數場景，既能保留邊界上下文，又不會造成大量重複內容
> 25%
不建議超過
相鄰 Chunk 出現大量重複文字，導致向量污染，浪費上下文空間
⚠ 向量污染（Vector Contamination）：Overlap 設太大（如 50%）時，幾乎每個 Chunk 都含有相同的文字，向量空間中大量重複向量會干擾相似度計算，Top-K 結果全是近似相同的 Chunk，白白浪費了 LLM 的上下文視窗。

⚡ 常見問題：上下文斷裂（Context Breakage）

PROBLEM — 代名詞跨 Chunk 斷裂示意
Chunk A
台灣的珍珠奶茶 起源於 1980 年代 由台南的茶飲店首創
Chunk B
它 後來傳播到全台灣 並在 2000 年後 開始向海外擴展
「它」指的是 Chunk A 中的「珍珠奶茶」，但被切入 Chunk B 後，LLM 在只讀 Chunk B 的情況下無法解析代名詞的指涉，導致語意斷裂。Overlap 只能緩解相鄰兩句的斷裂，跨越多句的代名詞參照難以靠 Overlap 解決。

🚀 進階技術：Late Chunking（延遲分塊）

✅ Late Chunking（Jina AI 提出）：先看再切
原始文件
→
全文 Embedding
→
再切 Chunk
→
區塊向量化
先對全文進行 Token 級別 Embedding，讓模型讀到整篇文件的上下文後，再進行分段，每個 Chunk 的向量都含有全局語意資訊。
✓ 保留全局語意，代名詞指涉更準確
⚠ 每個 Chunk 都含全局資訊，資訊密度較高，是一種取捨；建議在其他方法效果不佳時再嘗試

🔧 三個必須一起調的核心參數

✂️
Chunk Size
決定每段資料涵蓋多少內容，直接影響查準率與上下文完整性的平衡
🔗
Chunk Overlap
控制相鄰 Chunk 的重疊量，避免語意斷裂，但過大會造成向量污染
🧠
Embedding Model
決定語意理解能力與向量空間品質。不同模型對技術文件或自然語言的敏感度不同
這三個參數彼此互相影響，必須同時考量、反覆實驗，不能單獨調其中一個就認為最佳化完成。

💡 核心觀念

Chunking 沒有最佳解，只有透過實驗找到「可行解」。沒有任何公式能直接算出最佳 Chunk Size，唯有針對你的資料、你的查詢類型，設計實驗並量化評估，才能找到適合的參數組合。
沒有評估就是盲測。如果只是憑感覺設定 Chunk Size，而不去測量查準率與查全率的實際變化，那你只是在盲測，而不是在工程意義上的優化。
Embedding 決定語意理解能力，Chunking 決定語意保留品質。同樣的 Chunk，在不同的 Embedding 模型下，會產生完全不同的向量空間——這也是為什麼兩者必須一起考量與調教。
