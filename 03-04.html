🎯 最重要的觀念：找對比生成更重要

RAG 的重點不是生成模型有多強，而是找到生成的來源與條件。找到對的 Chunk，普通模型也能回答得不錯；找錯了來源，再強的模型也會胡說八道。

本單元從「資料工程」進入到「搜尋工程」，深入解析如何用最佳的算法策略找到最正確的參考資料。

📐 三種向量相似度計算方法

Cosine Similarity
餘弦相似度

計算兩個向量之間的夾角，只看方向、忽略向量長度。夾角越小（趨近 0°），Cosine 值越接近 1，代表越相似。

0°→1（完全相同）/ 90°→0（不相關）/ 180°→-1（相反）

✅ 語意搜尋最常用
L2 Distance
歐氏距離

計算兩個向量在向量空間中的直線距離。距離越小代表越相似，是傳統資料科學常用的距離計算方式。

缺點：容易受到向量長度（magnitude）影響，在不同尺度的資料上可能失真。

△ 部分資料庫預設使用
Dot Product
內積

計算兩向量的方向 × 強度的乘積，同時考量角度與大小。若向量已正規化（長度=1），結果等同 Cosine。

缺點：受向量大小影響，若未正規化，長度大的向量得分自然偏高。

△ 需正規化才適合語意搜尋
夾角小 → 相似度高
Cosine ≈ 0.97
兩向量方向幾乎相同
夾角大 → 相似度低
Cosine ≈ 0.17
兩向量方向差異大

⚡ 向量搜尋（Dense Retrieval）的優勢與限制

✅ 向量搜尋的優勢
  • 捕捉自然語言中複雜、模糊的表達（同義詞、近似詞、同音異字）
  • 理解文本之間的語意關係（上下位、因果、相似關係）
  • 跨語言搜尋：不同語言的向量也能互相理解與比對
  • 支援多模態搜尋：文字對圖片、文字對聲音
  • 不需要完全匹配關鍵字，語意理解後仍能找到相關資料
❌ 向量搜尋的限制
  • 精準關鍵字搜尋效果差：「第3條第2項」這類嚴謹用字容易失真
  • 數字、順序、代碼等結構性資料，向量化後語意會模糊
  • 越精準的匹配需求，向量搜尋反而越不適合
  • 法規條文、合約條款等需精確引用的場景，建議搭配關鍵字搜尋

BM25（Best Matching 25）— 傳統 TF-IDF 的進化版，ElasticSearch 的核心算法：
  • → 罕見詞（如專有名詞）權重提高，越獨特越重要
  • → 常見詞（你我他、的地得）權重降低，避免干擾
  • → 同時考量詞頻飽和度，避免同一個字重複出現導致分數虛高

🏗️ Multi-Stage Retrieval：兩階段策略

💡 單一階段搜尋難以同時兼顧 Recall（召回率） 和 Precision（精準率）。兩階段策略的核心思路：第一階段撈廣（高 Recall）→ 第二階段排準（高 Precision）。
01
🌊

初篩階段：快速大量召回（高 Recall）

用向量搜尋或 Hybrid Search 快速、便宜地從大量資料中撈出可能相關的候選集，寧可多抓也不要漏掉正確答案。資料量小可直接用向量搜尋，資料量大建議用 Hybrid。

目標：Top-20 ～ Top-100 策略：向量搜尋 / Hybrid 特性：快速、便宜
02
🎯

精排階段：Re-Rank 提升精準度（高 Precision）

對初篩的候選結果進行二次精細排序，判斷每個 Chunk 與 Query 的真實相關性，將最符合的資料往前推。直接對全量資料排序成本太高，因此先初篩再細排。

目標：Top-3 ～ Top-5 策略：Cross-Encoder Re-Rank 特性：準確、成本較高
03
🤖

生成階段：精排結果注入 LLM 上下文

將 Re-Rank 後排序最前的 Chunk 作為上下文，送入 LLM 生成最終回答。此時 LLM 拿到的是精準且相關的參考資料，回答品質自然提升。

輸出：準確且友善的回答 降低幻覺風險
為什麼要分兩階段？Recall vs Precision 的天生矛盾
RECALL（召回率）— 第一階段目標
把所有正確答案都撈出來，寧可多抓不要漏。向量搜尋天生 Recall 高，但排序不夠精準——答案可能在第6名而非前5名。
PRECISION（精準率）— 第二階段目標
取回的結果都是真正相關的，盡量減少雜訊。Re-Rank 模型對候選集做精細判斷，把真正相關的資料推到最前面。

✂️ Pre-Chunking vs Post-Chunking

📥 Pre-Chunking（靜態切分）

在資料存入向量資料庫之前就先切好，建立索引時資料形式已確定。

  • 資料穩定、容易維護
  • 容易 Scale，適合大量資料
  • 使用者問任何問題，都用同樣切好的資料回答（靜態）
📌 最常見的傳統做法
✓ 效率高、建置成本低
✗ 切錯了難補救（Embedding 已算好）
📤 Post-Chunking（動態切分）

先把整份文件存起來，等到查詢之後再依需求重新切分與整理。

  • 可根據問題、結果、上下文長度動態調整
  • 合併相鄰 Chunk、Sliding Window 補充上下文
  • 重新摘要成更精準的 Context 送給 LLM
🆕 近期流行的進階策略
✓ 回答品質更好，上下文更完整
✗ 查詢時處理成本較高
💡 混搭使用：實務上通常先用 Pre-Chunking 建立知識庫，再透過 Post-Chunking（如合併鄰近 Chunk、重新摘要）補足前者的不足，讓 LLM 能在生成前拿到最完整的上下文。

💡 核心觀念

找對比生成更重要。Retrieval 的目標是確保正確資料進入 LLM 的上下文。找對了，普通模型也能答得好；找錯了，再強的模型也會出錯。
單一搜尋方法難以滿足複雜查詢。向量搜尋擅長語意模糊匹配，BM25 擅長精準關鍵字；Hybrid Search 結合兩者長處，是現代 RAG 的推薦策略。
⚠ Multi-Stage 的本質是成本與品質的取捨。不是所有場景都需要兩階段——資料量小、查詢單純時，單階段向量搜尋即可。資料量大、需要高精準度時，才值得付出 Re-Rank 的額外成本。
