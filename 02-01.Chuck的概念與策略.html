為什麼需要 Chunking？
📐

LLM 的上下文視窗有限

無法把全部文本塞進 Context Window，必須切成小單位，讓 LLM 能有效參考。

🎯

資料太多反而讓 LLM 找不到重點

過多的參考資料會讓模型不知從何找起。精準的分塊才能讓 LLM 生成正確答案。

Chunk 的三重身份
Embedding 視角
語意最小單位
Embedding 模型向量化的基本單元，代表一段獨立的語意
資料庫視角
索引基本單位
向量資料庫中每筆記錄的粒度，決定搜尋的精細程度
Retrieval 視角
比對基本單位
使用者問題與資料庫進行相似度比對的最小粒度
影響 Chunk 品質的關鍵因素
📏

大小（Size）

Chunk 太小 → 資訊量不足，無法代表完整語意；Chunk 太大 → 噪音過多，資訊模糊，檢索精準度下降。

CHUNK SIZE
🔗

重疊（Overlap）

相鄰 Chunk 之間保留部分重疊文字，讓語意能夠延續，避免上下文在切割點被硬生生截斷。目標是維持語意連貫性。

OVERLAP
📊

檢索數量 K（Top-K）

每次從向量資料庫撈回與問題最相似的 K 個 Chunk，作為 LLM 生成答案的參考上下文，數量設定影響召回品質。

TOP-K
好的 Chunk vs. 壞的 Chunk
✅  好的 Chunk
能夠代表一句完整的話，語意自洽
能夠獨立回答一個問題，不依賴前後文仍可理解
包含完整的上下文，對齊人類對句子的理解
主題單一，不混入多個不相關概念
❌  壞的 Chunk
句子切一半——Embedding 模型無法理解這段在講什麼
一段內混著多個主題（A、B、C 混雜），向量失真
資訊量過多導致模糊，檢索精準度大幅下降
過短資訊不足，過長噪音過多，兩端都是壞的

「切得好，比切得多還要重要。」
Chunking 不是單純分段，而是語意的切分。每一塊都要有它的意義。
本集核心結論

🎯 五大帶走重點

1

RAG 四步驟：Extraction 抽取 → Splitting 分塊 → Embedding 向量化 → Loading 存入向量資料庫，Chunking 是第二步也是最關鍵的資料品質關卡。

2

決定 RAG 品質的不是模型，而是資料——目前模型已足夠強，問題幾乎都出在餵進去的 Chunk 品質好不好。

3

好的 Chunk = 語意最小單位——能代表一句完整話、能獨立回答問題、主題單一不混雜。

4

Overlap 重疊機制是關鍵設計，讓相鄰 Chunk 共享部分文字，避免語意在切割點斷裂。

5

好的參考資料 → 好的生成結果——就像學英文要給英文詞典，不是數學公式表。資料來源對了，LLM 才能生成正確答案。
