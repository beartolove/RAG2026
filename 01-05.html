📌 RAG 是什麼

Retrieval-Augmented Generation（RAG）= 搜尋檢索 + LLM 生成

RAG 的核心想法源自 Prompt Engineering 的延伸：既然 Few-Shot 可以給模型範例來引導回答，那麼也可以從外部資料庫動態檢索資料，補充進 Prompt 的上下文，讓 LLM 用它「本來不知道」的知識來生成答案。

簡而言之：讓很會說話的 LLM，加上隨時可擴充的外掛知識庫。

⚡ LLM 的兩大強項與侷限

⚠ LLM 的侷限

很會說話 ≠ 什麼都會

  • 知識量受訓練資料截止日限制
  • 缺乏企業內部或私有領域知識
  • 回答時效性問題時容易出錯
  • 最危險：看不懂卻用流暢語言生成錯誤答案（幻覺）
✓ LLM 的兩大強項

RAG 正是要善用這兩點

  • 表達能力強：能將資料整理成流暢、友善的回答
  • 豐富的通識知識：具備廣泛背景理解能力
  • RAG 把「找到正確資料」的責任交給檢索系統
  • 把「組織並呈現答案」的工作留給 LLM

🛠️ 優化 LLM 的三種方法對比

首選
✍️

Prompt Engineering

在問題前後加入指令、範例、引導語，讓 LLM 更精準地回答。

✓ 低成本、快速見效
⚠ 僅適用於「本來就會」的知識範圍
主力
🔍

RAG

動態從外部資料庫檢索相關內容，補充進 Prompt 上下文，彌補知識缺口。

✓ 適合時效性、私有、企業級知識
⚠ 需設計資料儲存與檢索流程
最後手段
⚙️

Fine-Tune 微調

以少量資料重新訓練模型，調整其行為或風格。

✓ 適合格式、風格、隱性規則調整
⚠ 難以用少量資料覆蓋大量訓練知識，正確性提升有限

💬 課程核心觀念

LLM 本質上是一種很會聊天的 AI，但「很會聊天」不等於「什麼都會」。RAG 就是幫它補上它不會的那些東西。
Fine-Tune 追求的是格式與風格；RAG 追求的是正確性與時效性。兩者目標不同，不應混為一談。
RAG 的品質關鍵不只是 LLM，而在於你怎麼找到對的資料、如何設計檢索流程，以及如何評估與調教每一個環節。
