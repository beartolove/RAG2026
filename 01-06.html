🎯 核心觀念：RAG 不是黑盒子

RAG 不只是一個 API 呼叫，而是由五個不同參數混搭組成的系統工程。
能夠理解並調教這五個要素，才能真正掌握 RAG 輸出品質的關鍵—— 而讓 RAG 結果變好的，不是換模型、改 Prompt，而是每一層環節的精準設計。

🧩 五大核心要素

COMPONENT 01
Chunking
資料切分
決定檢索的力度

將原始資料（PDF、文字、圖片、音訊等）分割成適當大小的片段，是整個 Pipeline 的起點。Chunk 切得太大或太小，都會嚴重影響後續檢索的準確性。

前端資料若沒切好，後面再怎麼優化，效果都相當有限。

chunk size overlap split strategy ⚠ 最容易被忽視的關鍵
COMPONENT 02
Retriever
資料檢索器
決定召回的範圍

將使用者的問題向量化後，從向量資料庫中找到語意最相似的 K 筆資料（Top-K Chunks），作為後續生成的上下文參考。

檢索策略可選擇純向量搜尋、關鍵字搜尋，或兩者混合的 Hybrid Search。

top-k vector search hybrid search index strategy
COMPONENT 03
Re-Ranking
重新排序
決定排序的品質

對 Retriever 召回的候選結果進行二次精排，讓更相關的片段優先放入 LLM 的上下文，進一步提升最終答案的準確性。

可選擇是否引用、是否壓縮、是否重新排列，依需求組合。

cross-encoder score threshold context compression
COMPONENT 04
Prompt
提示詞設計
決定生成的結構

設計如何將檢索到的 Context 組合進 Prompt，包含指令語氣、引用格式、輸出結構等。Context 怎麼放、以什麼方式引導 LLM，都直接影響答案的結構與可讀性。

system prompt context injection few-shot output format
COMPONENT 05
Generation
答案生成
決定輸出的風格

LLM 根據補充進來的知識與 Prompt 指令，生成最終回答。RAG 無法讓模型「更聰明」，但能讓模型擁有它本來不會的知識，結合其強大的表達能力產出準確而友善的答案。

LLM selection temperature citation style answer format

💡 RAG 黃金法則

前端決定後端。如果 Retrieval 沒做好、Chunk 沒切好，後面無論怎麼做，效果都非常有限。
RAG 無法讓模型變聰明，但能讓模型擁有它不會的知識。知識量的擴充 + LLM 的表達能力 = 正確且友善的回答。
讓 RAG 結果變好的，不是換模型、改 Prompt，而是 Chunk 怎麼設計、Embedding 怎麼選、Retrieval 策略是什麼、有沒有加上 Re-Rank、Context 怎麼組合。
