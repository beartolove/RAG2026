🎯 核心觀念：向量資料庫 ≠ 只是儲存工具

大多數人把向量資料庫只當成「存 Embedding 的地方」，但這只是工具層面的理解。真正的重點在於檢索——向量資料庫更像是一個搜尋引擎，它透過索引結構，讓 RAG 系統能快速從大量向量中找到最相似的結果，不需要一筆一筆精準比對。

🗂️ 資料庫類型定位

📋 RDS 關聯式資料庫

以結構化表格（Schema）儲存資料，支援 SQL 查詢，適合精準比對與關聯查詢。

  • MySQL、PostgreSQL
  • 結構嚴謹，查詢高效
  • 不適合非結構化資料
🌐 NoSQL 非關聯式資料庫

儲存非結構化資料，格式彈性（JSON、文件、Key-Value），適合大量異質資料。

  • MongoDB、Redis
  • 彈性高，擴充容易
  • 向量資料庫屬於此類
🔢 向量資料庫（Vector Database）— RAG 的核心記憶體

屬於 NoSQL 的一種，但專門儲存向量（Embedding 編碼後的結果），並以索引結構支援近似向量搜尋（ANN）。

它是一個具有索引結構的集中儲存系統：存的不是欄位，而是向量；查的不是精準字串，而是語意相似度。只要能被 Embedding 的資料（文字、圖片、音訊），都可以存入向量資料庫供 LLM 參考。

⚠ 注意：RAG 不需要 100% 精準的索引。向量搜尋的目的是「模糊語意比對」，過度精準反而失去了語意搜尋的靈活性。IVF 和 HNSW 在保留模糊搜尋能力的同時，提供更好的速度。

⚖️ 選擇時的三個核心取捨維度

📦
DATA SCALE

資料量大小

資料量小 → 本地端即可；中大型 → 考慮 SaaS 或自架；超大規模 → 必須自架並優化 Index

💰
COST

成本預算

Prototype 用免費本地端；封測上線可用 SaaS；需長期控制成本則選自架方案

⏱️
LATENCY

可接受的延遲

對回應速度要求高的場景，需選擇 HNSW 索引的資料庫，並考量網路延遲（本地 vs 雲端）

💡 核心觀念

向量資料庫是 RAG 的長期記憶，而非只是倉庫。LLM 本身沒有長期記憶，只能看當下的 Context Window；資料庫才是讓 LLM 具備「查歷史、做搜尋」能力的關鍵組件。
Index 決定了向量資料庫快不快、準不準、記憶體佔多少。作為使用者，需要了解不同資料庫預設使用的索引機制，才能在精準度、速度與成本之間做出正確取捨。
好的向量資料庫選擇，決定了 RAG 能否快速 Scale。初期 Prototype 用本地端，封測上線用 SaaS，最終生產環境需根據資料量、延遲需求與控制需求選擇自架方案。
