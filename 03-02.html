向量資料庫與嵌入式模型比較

張維元 ── RAG 精準搜尋實戰 03-02 | Embedding 決定語意品質，向量資料庫決定搜尋效率，RAG 準不準由診斷 Pipeline 決定

⚡ 最重要的一句話：兩者的職責分工

🧠
EMBEDDING MODEL

嵌入式模型

決定語意品質

負責把文字轉換成有意義的向量，決定向量空間「長什麼樣子」——相似的概念靠近，不相似的遠離。語意空間設計的好，LLM 才能有效找到適合資料。

  • 文件 Embedding → 向量存入資料庫
  • Query Embedding → 向量比對搜尋
  • 模型好壞決定「找到的東西有沒有意義」
⟺
🗄️
VECTOR DATABASE

向量資料庫

決定搜尋效率

負責在高維向量空間中，用索引結構快速找到最接近的向量。資料庫設計好壞決定「找得夠不夠快」，但無法改善語意本身的品質。

  • 索引結構決定查詢速度
  • 不同 Index 在速度 / 精準 / 記憶體間取捨
  • 資料庫解決效率問題，不解決語意問題

🔄 Embedding 在 RAG 流程中出現兩次

📥 第一次：索引階段（離線）
將文件切成 Chunk 後，把每個 Chunk 轉換成向量，存入向量資料庫，建立語意索引。這一步決定了知識庫的「語意地圖」。
🔍 第二次：檢索階段（即時）
使用者發出查詢時，把 Query 也轉換成向量，再和資料庫中的向量計算相似度，取出最近的 K 筆 Chunk 作為上下文。
💡 兩次必須使用同一個 Embedding 模型，才能確保向量空間對齊，讓 Query 向量和 Chunk 向量能正確比對。

📐 從 One-Hot Encoding 到 Word Embedding

❌ 傳統做法：One-Hot Encoding（稀疏矩陣）

用一個超長的 0/1 向量代表每個字，每個字只對應一個位置為 1，其餘全為 0。

apple → [1, 0, 0, 0, ...]
dog → [0, 1, 0, 0, ...]
cat → [0, 0, 1, 0, ...]

只能表示「這個字有沒有出現」，無法表示詞義的遠近。

⚠ apple 與 dog 的距離 = apple 與 elephant 的距離
⚠ 完全沒有語意概念，無法做語意搜尋
✅ Embedding：連續向量空間（稠密矩陣）

每個詞被映射到一個連續的低維向量，透過神經網路學習大量文本，保留詞與詞的語意關係。

dog → [0.72, -0.31, 0.55, ...]
cat → [0.68, -0.28, 0.51, ...]
plane→ [-0.12, 0.89, -0.34, ...]

語意相近的詞，向量距離也相近，支援語意運算。

✓ dog 與 cat 距離 < dog 與 plane 距離
✓ 語意相似度 = 向量空間距離

🕰️ Embedding 技術的發展脈絡

2013

Word2Vec（Google）— Embedding 的起點

利用神經網路從大量文本學習詞與詞之間的關係，讓每個詞都能變成一組有意義的向量，且向量能保留原本文本間的語意關係。這是整個 Embedding 技術的奠基之作。

Token

模型看的不是「字」，是「Token」

Token 是模型視角中能讀懂的最小語意單位——有時是一個字，有時是半個字，有時是一個詞。語言模型本質上在做機率預測：根據前面的 Token，預測下一個最可能出現的 Token。

今日

Sentence / Document Embedding

從單詞延伸到句子、段落整體的向量化，讓 RAG 能以整段 Chunk 為單位進行語意搜尋，是現代 RAG 系統的核心技術基礎。

💡 語意空間設計的好，就是讓「語意相似的概念向量靠近，不相似的遠離」。這讓電腦能用數學運算來理解人類語言——語言從此不只是文字，而是一個數學結構。
⚡ Word2Vec 經典示範：向量可以做語意運算
King − Man + Woman ≈ Queen
代表語意變成了向量運算。電腦不懂文字意義，卻能透過向量之間的關係進行「語意推理」——這正是整個 RAG 能夠成立的數學基礎。

⚠️ 換 Embedding 模型，不一定讓 RAG 變準

「Embedding 好 ≠ RAG 準」——影響 RAG 準確率的是整條 Pipeline，而不只是 Embedding 模型的品質。換了更好的模型，若前面的 Chunk 切壞了，照樣沒有救。
三種即使換好 Embedding 也救不了的情況
情況一：Chunk 切壞
Chunk 的語意被切得零散，再強的 Embedding 模型也無法從破碎的片段中提取有意義的向量。
情況二：Top-K 沒包到答案
Retrieval 只抓了前 3 筆，但答案在第 4 筆。Embedding 再好，沒有被取回就沒有用——這是 Top-K 設定的問題。
情況三：Index 設計過於激進
Index 的召回策略設計不當，導致真正最近的向量反而找不到，問題出在資料庫配置層，不是 Embedding 層。
✅ 解法：診斷 Pipeline——找到究竟是哪一層出了問題，再針對性調整。

🏆 RAG 調優的優先順序

01

Chunk（資料切分）

最根本、影響最大。Chunk 切壞，後面一切都救不了。語意單元的完整性決定了整條 Pipeline 的上限。

↓
02

Retrieval（檢索策略）

Top-K 數量、混合搜尋策略、是否加入 Re-Rank。決定答案有沒有被找到。

↓
03

Embedding Model（嵌入模型）

決定語意空間品質。不同模型對技術文件、自然語言、中文的敏感度不同，需實驗選擇。

↓
04

Index（索引結構）

決定搜尋速度與記憶體用量。在前三層都設計良好的前提下，調整 Index 可進一步提升效能。

💡 核心觀念

Embedding 不只是 API，是語意空間的設計問題。好的 Embedding 讓「全世界的文字壓縮在一個可被數學運算的向量空間」，語意越對齊，RAG 找到的資料就越有意義。
語言不再只是文字，而是一個數學結構。只要語意能被向量化，就能做距離的相似性搜尋——這是整個 RAG 能夠成立的根本原因。
RAG 準不準，由診斷 Pipeline 決定，不是只靠換模型。優先序：Chunk → Retrieval → Embedding → Index。每個環節都做好，效果自然水到渠成。
