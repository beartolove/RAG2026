搜尋技術發展演進

完整梳理搜尋技術從 1960 年代關鍵字比對，演進至語意搜尋，再到現代 RAG 生態系的脈絡

張維元 · RAG 精準搜尋實戰 01–02
搜尋的本質

搜尋的核心目標：從大量資訊中，快速找到對使用者有價值的內容

衡量一個搜尋系統的關鍵，永遠只有兩件事：查詢速度與結果準確性。整條搜尋技術線可追溯回 1960 年代，歷經三個世代的演進，才走到今日的 RAG 架構。

三大演進階段
1960s →
第一代
📚

Information Retrieval (IR) 關鍵字比對

  • 像查字典：輸入關鍵字，找出出現該詞彙的文件位置
  • 核心技術：Indexing 索引——如 Google 的 Inverted Index（反向索引）
  • 整個流程：儲存 → 索引 → 解鎖 → 排序
  • 限制：要求字串完全匹配，無法理解語意或意圖
  • 來自資料庫領域，本質是解決「資料太多，如何快速查找」的工程問題
2010s →
第二代
🧠

Semantic Search 語意搜尋 向量相似度

  • 解決「模糊搜尋」需求：你要搜的概念不一定是明確關鍵字
  • 例：搜尋「減脂」→ 語意擴展為「飲食控制、有氧運動、熱量限制」
  • 核心技術：Embedding 向量化——讓電腦理解文字的語意關係
  • 從精準比對 keyword → 概念匹配的 Semantic 向量空間
  • 限制：只能「找到」文件，無法讓結果變得更易讀、更友善
2022 →
第三代
⚡

RAG — Retrieval Augmented Generation 檢索 + 生成

  • 將語意檢索（找資料）與 LLM 生成（產出答案）串連為一體
  • 把排序升級為 ReRank，把搜尋升級為向量相似模糊搜尋
  • RAG 不是取代搜尋，而是讓搜尋結果更智慧、更 User-Friendly
  • 讓 LLM 不只是聊天機器人，而是企業等級的知識庫智慧系統
  • RAG 是系統工程：可拆解、可調教、可逐層優化
現在 →
進階發展
🚀

Agentic RAG / GraphRAG 主動解鎖

  • Hybrid Search：關鍵字搜尋 + 語意搜尋的混合策略
  • GraphRAG：以關係結構重新組織知識，找到更有關聯性的資料
  • Agentic RAG：主動解鎖，讓系統自行決策如何搜尋
  • 加入記憶、結構化／非結構化資料整合，擴展成完整生態系
核心概念解析
🗂️

Inverted Index 反向索引

IR 的核心技術，記錄每個關鍵字出現在哪些文件哪個位置，實現毫秒級查詢。Google 搜尋引擎的基礎。

🔢

Embedding 向量化

將文字轉換為高維度數值向量，讓電腦能夠計算語意相似度。是語意搜尋與 RAG 的共同基礎。

🔃

ReRank 重新排序

RAG 對傳統排序的升級。在召回候選文件後，再用更精細的模型重新評分排列，提升結果品質。

🏗️

RAG 是系統工程

可拆解為 Pre-retrieval、Retrieval、Post-retrieval 多層。每層都能獨立優化，從 Prototype 到 Production 的關鍵。

🏢

企業知識庫應用

從 BI 報表到自然語言問答，RAG 讓 Data Lake / Data Warehouse 能夠被語意查詢，打造企業等級智慧中台。

🔧

調教能力的重要性

知道結果不好只是起點，真正的能力是反推哪一層出了問題並修正——就像資深修車師傅診斷引擎聲。

三代技術對比
技術 搜尋方式 核心技術 優勢 限制
IR 資訊檢索 關鍵字精準比對 Inverted Index 速度快、精準 無法理解語意、須完全匹配
Semantic Search 向量相似度模糊搜尋 Embedding 向量化 理解意圖、模糊擴展 結果不易讀、缺乏生成能力

課程核心結論

🎯 本集五大重點

1

理解脈絡才能用好工具——RAG 不是憑空出現的魔法，是 60 年搜尋技術演進的自然結果。

2

RAG 不取代搜尋，而是升級搜尋——它在傳統架構上，把排序升為 ReRank、把比對升為語意向量搜尋，再加上 LLM 生成。

3

Embedding 是語意搜尋的核心——讓電腦能夠理解「減脂 ≈ 飲食控制 + 有氧運動」這種語意關聯。

4

RAG 是系統工程，不只是 Prompt Engineering——設計資料來源、搜尋策略、評估、優化才是 RAG 的真正挑戰。

5

目標是 Production 等級——從 Prototype 到真正可用、好用且能持續進步的 RAG 系統，需要完整的工程能力。
